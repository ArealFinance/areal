# Operator Runbook — Full Contract Redeploy (Fornex VPS test-validator)

**When to use:** any time you wipe the VPS test-validator ledger and redeploy all
contracts from scratch (fresh mints + fresh bootstrap). Follow this end-to-end —
several surfaces hardcode or cache mint addresses and silently break after a reset
if skipped. Every gotcha below was a real failure caught in the 2026-05-18 redeploy.

> Identity (keypairs, program IDs) — see memory `reference_vps_deploy_identity`.
> Authority = `CRAKiyq2eau8fiNxH2DMhTzry2Jn3kKmXnhZGnbdyFvp` (deployer, local + VPS
> `deploy-keypair.json`). Program vanity IDs are pinned — they DON'T change on reset.
> Only **mints rotate** (RWT/USDC/SPRK) — that is the entire source of post-reset breakage.

---

## What is automated vs manual

`deploy-fornex.sh` (after commit `853442f`) now handles: contract build+deploy, R20
mint-pin, bootstrap (3 tokens + 2 pools + YD distributor), **bots build**, and
**crank systemd service install**. Everything else below is **manual** and is the
part people forget.

## 0. Pre-generate mint keypairs (REQUIRED before deploy)

`deploy-fornex.sh` step 4.6 (R20 mint-pin) reads `data/e2e-bootstrap.secrets.json`.
If absent it FAILS. Generate RWT+USDC keypairs locally with the **`rwt < usdc`
byte-order invariant** (master pool needs USDC on `vault_b` for CP-7 grow_liquidity):

```bash
# Easiest: run verify-fresh-deploy.sh's stage_pregen_keypairs, OR generate inline.
# Inline (mac): keep regenerating RWT until its bytes sort < USDC bytes.
# Writes data/e2e-bootstrap.secrets.json with rwt_mint_* + usdc_test_mint_*.
```
The full inline snippet is in this session's history; the canonical source is
`scripts/verify-fresh-deploy.sh::stage_pregen_keypairs` (lines ~138-230). SPRK is
generated by bootstrap itself — no pre-gen needed.

## 1. Reset the VPS validator

```bash
ssh vps-vpn 'systemctl stop solana-test-validator; \
  rm -rf /root/test-ledger /opt/areal/data/test-ledger \
         /opt/areal/data/e2e-bootstrap.json \
         /opt/areal/data/e2e-bootstrap.secrets.json \
         /opt/areal/data/r20-migrated.json; \
  systemctl start solana-test-validator; sleep 8'
# Airdrop to the deployer (CRAKi) on the fresh ledger:
ssh vps-vpn 'solana airdrop 1000 CRAKiyq2eau8fiNxH2DMhTzry2Jn3kKmXnhZGnbdyFvp --url http://localhost:8899'
# Fix git ownership (earlier root ops can leave objects unwritable by deployer):
ssh vps-vpn 'chown -R deployer:deployer /opt/areal && git config --global --add safe.directory /opt/areal/bots'
```
Also clear the matching local artifacts: `rm -f data/e2e-bootstrap.json data/r20-migrated.json` (keep the secrets you just generated in step 0).

## 2. Full deploy

```bash
./scripts/deploy-fornex.sh --force-keypair
```
- `--force-keypair` pushes the local `deploy-keypair.json` (CRAKi) to the VPS.
- **Watch for:** `Summary: 4/4 GREEN` smokes. Step 10 prints a **"SDK localnet mints
  out of sync"** warning — that is expected; it's your cue for step 3.
- Verify 5 programs deployed at low slots (fresh ledger) and bootstrap timestamp is now.

## 3. Sync SDK localnet mints (MANUAL — frontend/backend depend on this)

Without this, the frontend reads dead mints and every Price/TVL/Supply/Holders shows `—`.

```bash
./scripts/sync-sdk-localnet-mints.sh   # rewrites RWT_MINTS.localnet + USDC_MINTS.localnet, bumps SDK
```
Then **repin SPRK manually** (the OT mint is NOT in the SDK mint tables):
```bash
SPRK=$(ssh vps-vpn 'jq -r .ots[0].ot_mint /opt/areal/data/e2e-bootstrap.json')
# edit app/src/lib/swap/pool-catalogue.ts → const SPRK_MINT = new PublicKey('<SPRK>')
```
Commit + push **sdk** and **app**, then bump submodule pointers in the meta-repo and push.

> The app's `endpoints.ts` reads `USDC_MINTS.localnet` / `RWT_MINTS.localnet` from the
> SDK (no hardcoded overrides — fixed 2026-05-18). So the SDK bump is enough for RWT/USDC;
> only SPRK needs the manual pool-catalogue repin.

## 4. Redeploy the frontend

```bash
ssh vps-vpn /usr/local/sbin/areal-deploy-app
```
Verify a fresh chunk carries the NEW pubkeys (Cloudflare caches immutable assets —
the chunk hash changes when content changes):
```bash
for c in $(curl -s https://app.areal.finance/ | grep -oE '/_app/immutable/chunks/[A-Za-z0-9._-]+\.js' | sort -u); do
  curl -s "https://app.areal.finance$c" | grep -oE '<NEW_RWT>|<NEW_USDC>' | sort -u && echo "  ^ $c"
done
```

## 5. Backend faucet (Docker — MANUAL rebuild + env)

`faucet.constants.ts` reads USDC from the SDK (auto after rebuild), but the faucet's
**signing keypair env** must point at the new USDC mint authority (= deployer CRAKi).

```bash
# On VPS: pull new backend code (meta submodule bump) + refresh env + rebuild container.
ssh vps-vpn 'cd /opt/areal && git pull -q origin main && git submodule update --init backend'
# Set FAUCET_USDC_AUTHORITY=CRAKi + refresh FAUCET_USDC_AUTHORITY_KEYPAIR_B64 from the
# deployer keypair (generate b64 ON the VPS — never echo secrets):
ssh vps-vpn 'cd /opt/areal/backend && python3 - <<PY
import json,base64,re,os
b64=base64.b64encode(bytes(json.load(open("/root/.config/solana/deploy-keypair.json")))).decode()
ls=open(".env").read().splitlines()
def setk(ls,k,v):
    out=[]; f=False
    for l in ls:
        if re.match(rf"^{re.escape(k)}=",l): out.append(f"{k}={v}"); f=True
        else: out.append(l)
    if not f: out.append(f"{k}={v}")
    return out
ls=setk(ls,"FAUCET_USDC_AUTHORITY","CRAKiyq2eau8fiNxH2DMhTzry2Jn3kKmXnhZGnbdyFvp")
ls=setk(ls,"FAUCET_USDC_AUTHORITY_KEYPAIR_B64",b64)
open(".env","w").write("\n".join(ls)+"\n"); os.chmod(".env",0o600)
PY'
ssh vps-vpn 'cd /opt/areal/backend && docker compose -f docker-compose.prod.yml up -d --build backend'
```
Smoke: `curl -s -w '\n%{http_code}' https://api.areal.finance/faucet/usdc -H 'authorization: Bearer <JWT>' -H 'content-type: application/json' --data-raw '{"wallet":"<wallet>"}'` → expect `200` + tx sig.

## 6. Bots + rewards

`deploy-fornex.sh` now builds bots + installs the 5 crank services. Verify:
```bash
ssh vps-vpn 'for s in merkle-publisher revenue-crank convert-and-fund-crank yield-claim-crank nexus-manager; do printf "%-28s %s\n" $s "$(systemctl is-active areal-$s.service)"; done'
```
All must be `active`. The revenue→convert→fund pipeline then funds the YD reward vault
on its own. For a demo with **immediately visible** rewards, seed the vault and ensure
the demo wallet holds **≥100 OT** (`MIN_HOLDING_OT_LAMPORTS` — below it = excluded from
the merkle tree = 0 RWT/day, regardless of pipeline health).
